## 1. Wins

- **Turned a contested core-PR into a clean external-repo + small docs-PR** — alfred's review on [#970](https://github.com/Fission-AI/OpenSpec/pull/970) signaled "external is better"; the response shipped a fully-working v1 community schema + a 21-line docs PR ([#1043](https://github.com/Fission-AI/OpenSpec/pull/1043)) instead of fighting for built-in inclusion. PR #970 close comment listed the three concerns and how each was concretely addressed.
- **Ate own dog food end-to-end** — used `superpowers:brainstorming` (validate plan), `superpowers:writing-plans` (Phase 1 plan), `superpowers:subagent-driven-development` (Phase 1 execution: 9 implementer + spec/quality reviewers per task + final whole-impl reviewer). The schema-being-built was used to drive its own build.
- **Fact-check culture held under pressure** — when user pushed back on the "2b loses TDD" claim ("感覺有矛盾"), the response was to read `superpowers:executing-plans` SKILL.md verbatim and verify, not to defend the original framing. Result: stronger evidence (`grep` proof) + the right structural fix (commit `045ab98` drops 2b entirely).
- **Bilingual EN canonical + zh-TW from the start** — every public-facing doc has both languages with toggle links. Internal docs (CLAUDE.md, retrospectives) stay zh-TW as maintainer-facing. No mixed-language single files.
- **17 commits, atomic + reviewable** — each commit corresponds to one logical change (baseline import → LICENSE → manifest → remove auto-commit → skill PRECHECK → retrospective artifact → evidence PRECHECK → rename → docs → CI → review polish → bilingualization → consolidation → drop 2b). `git log --oneline` reads like a changelog.
- **Final code-review caught 4 Important issues** that would have undermined the PR #970 close-comment narrative: stale INTEGRATION section 3-0, retro PRECHECK pattern that never matched verify.md template, stale README workflow diagrams, missing skill-PRECHECK section in INTEGRATION. All fixed before push to remote.

## 2. Misses

- 🟡 **2b dead-code only caught by user, not by initial design review** — Step 0 PRECHECK required `superpowers:subagent-driven-development`, which would block the very platforms 2b was supposed to serve. Should have caught at design-time. Self-review on the plan didn't surface this; user did during fact-check round (commit `045ab98`).
- 🟡 **Bridge README and INTEGRATION had ~60% content overlap** that survived multiple review passes — only surfaced when user explicitly asked "find redundancy". Required a 1168-line cleanup commit (`862e163`). Should have been caught at the writing-plans review stage; instead, reviewer agents focused on per-task correctness, not cross-file coherence.
- 🟡 **First push to GitHub failed** due to OAuth token missing `workflow` scope. Could have been anticipated when planning to push a `.github/workflows/*.yml` file. Cost: one user round-trip to refresh scope.
- 📌 **T8 implementer left orphan `## What it does` heading** because the task spec's `new_string` ended with that heading expecting a body that wasn't written. Caught in spot-check, fixed via `git commit --amend`. Plan should have included the body.
- 📌 **Final whole-implementation code-reviewer hit Stream idle timeout** (~7 min, 60 tool uses) on user's client. Got partial response. Subagent finished correctly server-side, but UX was poor. Should have either split into smaller reviewers or run in background mode.
- 📌 **Initial `extension.yml` shipped without challenging whether OpenSpec actually reads it** — only when user asked "what's it for?" did the analysis surface that it's pure documentation contract. Eventually deleted in `862e163`. Could have been deleted at design time if the question had been asked earlier.

## 5. Surprises

- **spec-kit ecosystem already had 2 mature superpowers bridges** ([RbBtSn0w/spec-kit-extensions/superpowers-bridge](https://github.com/RbBtSn0w/spec-kit-extensions/tree/main/superpowers-bridge), [WangX0111/superspec](https://github.com/WangX0111/superspec)) before Phase 0 even started. Initial plan assumed we were forging a path; reality is we joined a small ecosystem with established patterns. The research changed the path A vs path B framing significantly.
- **`executing-plans` SKILL.md is dramatically thinner than `subagent-driven-development`** (~50 lines vs ~280, no TDD or code-review references at all). The "fallback" framing in the original schema implied feature parity; verbatim reading proved otherwise.
- **The OpenSpec engine literally cannot enforce `verify.requires: [plan]` timing** — schema graph is filesystem-state-based, not state-machine-based. Layer-2 evidence-based PRECHECK (concrete `git log` / `grep` against runtime state) is a workable mitigation, but the right fix is engine-level support (`post_apply` phase analogous to spec-kit's `after_implement`).
- **Top-level README and bridge README served different audiences but their existence wasn't obvious until the consolidation audit.** Top README is "what's in this repo"; bridge README is "this specific bridge". Both useful; could only be consolidated within the bridge's own files (README + INTEGRATION), not across that boundary.
- **OAuth `workflow` scope is a thing** — push got rejected because the token couldn't create `.github/workflows/*.yml`. Asking the user to refresh scope was a one-time cost; worth remembering for any new repo with CI on first push.

## Verdict

End-to-end design → execute → response cycle worked. The pivot from "modify PR #970 in-place" to "external repo + small docs PR" added ~2 days of work but produced a much cleaner narrative for alfred and a reusable foundation for future community schemas. Three Misses (2b dead-code, doc overlap, push scope) were cheap to recover from but reveal a gap: design review didn't catch cross-cutting consistency. The fact-check moment around 2b was the highest-value single intervention.

Recommended next change: **add a "design-review pre-commit checklist"** to `CLAUDE.md` covering (1) doc overlap, (2) schema dead-code paths, (3) extension.yml-style "is this actually used" challenges. The Promote candidates table above lists everything worth migrating.
